## questions:
Why avoid `latest` tags?
>Traceability and rollback clarity — SHA-only tagging lets you pinpoint exactly which commit was deployed and easily revert to a known-good state.

Does GitHub Actions runner RAM affect the 4GB VPS budget?
>No. GitHub runners execute on GitHub's infrastructure; the VPS only receives the container image. Runner resources are GitHub's problem.

Should we pin ubuntu-latest to 22.04?
>No. Keep `ubuntu-latest` — it's stable enough and simpler than managing explicit version pins across future GitHub updates.

## tradeoffs, decisions why:
Combine D + E into single workflow job
>Ship faster with everything wired in one commit. Still safe because we fixed all issues in code review before pushing.

Derive SHA from build job output instead of git
>Ensures deploy uses the exact image that was built and pushed, eliminating race conditions between build and deploy.

Require explicit kubectl version
>Prevents breaking changes from silent upgrades; 1.27+ is stable for our k3s version.

## notes
**SHA race condition**
>If deploy re-derives SHA independently, build might have pushed a different image (e.g., different git state). Using build job's output SHA guarantees consistency.

**sed pattern scoping**
>`:SHA` matches `:` + literal string; must be full `image: nginx:SHA` to avoid false matches in other fields (env vars, etc).

**Kubeconfig persistence**
>Base64 secret decoded and written to disk; must clean up on failure to avoid leaking secrets in runner logs.

**Rollout verification**
>`kubectl rollout status` blocks until ready or timeout. Failure here is catchable, unlike silent deployment lag.
